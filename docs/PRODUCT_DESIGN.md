## 1. 产品定位

SpeakSum 是一个**个人会议知识管理与思维可视化平台**。

它不仅仅是会议纪要工具，更是帮助用户：
- 从杂乱会议转写中提取有价值的个人思考
- 自动构建个人知识图谱，可视化展示话题关联与思维演变
- 生成可消费的结构化数据，为 AI Agent 训练提供语料

**核心价值主张**：
> "让每一次会议发言都成为你知识图谱的一个节点"

## 2. 用户画像

### 主要用户：知识工作者

**小美，产品经理**
- 每周参加 5-8 场会议，经常需要回顾之前的决策思路
- 痛点：散落在各处的会议记录，无法追踪自己对某个话题的观点变化
- 使用场景：上传飞书会议转写 → 查看知识图谱中"用户增长"话题的演变 → 发现 3 个月前的观点与现在不同

**阿强，技术负责人**
- 需要沉淀技术决策的思考过程，用于团队分享和复盘
- 痛点：技术方案讨论过程散落，难以系统整理
- 使用场景：批量上传往期技术评审会议纪要 → 生成架构决策知识图谱 → 导出给团队学习

**Lisa，AI 应用开发者**
- 希望训练一个了解自己的 AI Agent
- 痛点：没有结构化的个人发言数据
- 使用场景：持续积累会议数据 → 导出 JSON 语料 → 用于微调个人 AI 助手

### 3.2 详细流程说明

#### 流程 1：首次使用

1. **欢迎页** - 简洁的产品价值说明，CTA 按钮「上传第一份会议纪要」
2. **文件上传** - 拖拽或选择文件，支持 .txt / .md / .doc / .docx
3. **配置确认** - 选择说话人（从常用身份选或临时添加），确认会议时间/标题（自动识别，可修改）
4. **处理等待** - 实时进度条，显示当前处理阶段
5. **结果展示** - 直接进入「会议时间线」视图，展示处理后的第一条会议
6. **引导探索** - 提示用户切换到「知识图谱」视图查看关联

#### 流程 2：日常使用

1. **首页入口** - 展示「会议时间线」和「知识图谱」两个卡片，显示最新数据摘要
2. **快速上传** - 顶部常驻上传按钮，支持批量选择多个文件
3. **即时反馈** - 上传后立即在时间线中看到新会议条目（处理中状态）
4. **后台处理** - 用户可继续浏览其他内容，处理完成通过红点/通知提示
5. **增量更新** - 新会议自动融入现有知识图谱，无需手动刷新

#### 流程 3：知识探索

1. **进入图谱** - 点击「知识图谱」入口，加载个人知识全景
2. **探索岛屿** - 看到多个话题聚类（岛屿），岛屿大小反映话题讨论频次
3. **深入话题** - 点击某个岛屿，展开时间轴视图，显示该话题下的所有发言
4. **发现关联** - 看到岛屿之间的连线，点击连线了解关联原因
5. **时间穿越** - 在话题内滑动时间轴，观察观点随时间的演变
6. **溯源查看** - 点击任意发言，查看原始会议上下文

## 4. 功能优先级 (Functional Priority)

### P0 - 必须有 (MVP)

| 功能 | 描述 | 验收标准 | 业务价值 |
|------|------|----------|----------|
| **文件上传** | 支持 .txt/.md/.doc/.docx 上传 | 拖拽上传，格式自动识别，最大 10MB | 核心入口 |
| **发言提取** | 自动识别并提取目标说话人发言 | 准确率 > 90%，支持自定义说话人名称 | 核心功能 |
| **口语清理** | 去除语气词、修正错别字 | 清理后文本通顺，保留原意 | 质量保证 |
| **Markdown 报告** | 生成人类可读的会议纪要 | 包含时间线、原文、清理后文本、金句 | 主要输出 |
| **知识图谱可视化** | 话题聚类岛屿 + 时间排列 | 自动布局，支持缩放/拖拽/点击 | 核心卖点 |
| **双主页切换** | 会议时间线和知识图谱两个视图 | 切换流畅，数据同步 | 产品形态 |
| **用户认证** | 注册/登录/找回密码 | 邮箱或第三方登录 | 数据隔离 |
| **模型配置** | 支持多模型供应商配置 | 默认 Kimi 2.5，可配置其他 | 灵活性 |

### P1 - 应该有 (V1.0)

| 功能 | 描述 | 验收标准 | 业务价值 |
|------|------|----------|----------|
| **金句提炼** | 自动提取核心观点并书面化 | 每段发言 1-3 条金句，可编辑 | 价值提升 |
| **话题标签** | 自动为发言打标签 | 支持自动+手动，标签可合并 | 组织管理 |
| **批量上传** | 一次上传多个文件 | 最多 50 个，队列处理 | 效率提升 |
| **历史会议管理** | 列表展示、搜索、删除 | 按时间排序，支持关键词搜索 | 数据管理 |
| **JSON 导出** | 导出结构化数据 | 完整 Schema，可用于 AI 训练 | AI 场景 |
| **图谱筛选** | 按时间/话题/关键词筛选 | 实时过滤，动画过渡 | 探索体验 |
| **处理进度实时展示** | 显示当前处理阶段 | 解析→提取→清理→生成→图谱 | 透明度 |
| **智能分块** | 长文本自动分块处理 | 5万字会议也能处理 | 大文件支持 |

### P2 - 可以有 (V1.5)

| 功能 | 描述 | 验收标准 | 业务价值 |
|------|------|----------|----------|
| **话题关联手动调整** | 添加/删除话题间关联 | 简单易用的关联编辑器 | 准确性 |
| **时间对比模式** | 对比两个时间点的观点 | 并排展示，高亮差异 | 深度分析 |
| **导出 PDF** | 生成精美的 PDF 报告 | 保留样式，适合打印分享 | 场景扩展 |
| **邮件摘要** | 定期发送知识图谱摘要 | 每周/每月自动推送 | 用户留存 |
| **API 接口** | 提供 REST API | 支持第三方集成 | 生态 |

### 5.3 交互设计原则

#### 原则 1：渐进式披露
- 首次用户：简洁引导，突出核心功能
- 活跃用户：快速入口，高效操作
- 深度用户：高级筛选，批量操作

- 时间线中的每条发言，都可「在图谱中查看」
- 图谱中的每个节点，都可「查看原始会议」
- 两个视图始终保持数据关联

- 上传后立即显示在处理队列
- 处理进度实时更新
- 图谱变化平滑动画过渡

#### 原则 4：容错设计
- 文件格式错误时，明确提示支持的格式
- 解析失败时，显示原始内容允许手动调整
- 处理中断时，支持断点续传

#### 原则 5：智能默认
- 自动识别说话人（从常用列表匹配）
- 自动识别会议时间/标题（从文件名/内容）
- 自动分块处理长文本（用户无感知）

## 7. 计算规则与算法逻辑

### 7.1 日期识别算法

#### 识别优先级（从高到低）

```
1. 文件名匹配（正则表达式列表）
   
   模式 1: 完整日期
   - 正则: (\d{4})[-_./]?(\d{2})[-_./]?(\d{2})
   - 匹配: "2024-04-01_meeting.txt", "20240401.docx", "product.2024.04.01.md"
   - 提取: 年=2024, 月=04, 日=01
   
   模式 2: 简写日期（当年）
   - 正则: (\d{2})[-_./]?(\d{2})
   - 匹配: "0401_meeting.txt", "meeting_0401.docx"
   - 处理: 推断为当年，如果推断日期 > 今天，则推断为去年
   
   模式 3: 中文日期
   - 正则: (\d{4})年(\d{1,2})月(\d{1,2})日
   - 匹配: "2024年4月1日会议纪要.txt"

2. 文件内容第一行匹配
   
   模式 1: 标题行日期
   - 正则: ^.*?(\d{4})[-/](\d{1,2})[-/](\d{1,2}).*?$
   - 匹配: "产品策略会 - 2024-04-01", "2024/4/1 会议记录"
   
   模式 2: 会议信息行
   - 正则: 时间[:：]\s*(\d{4})年
   - 匹配: "时间：2024年4月1日"

3. 文件元数据
   - docx: 从 document.xml 读取创建时间
   - md: 从 YAML frontmatter 读取 date 字段
   - txt: 使用文件系统创建时间（最后手段）

4. 无法识别
   - UI 状态: 显示日期选择器，标记为 ⚠️ 必填
   - 快捷选项: 今天、昨天、本周一、上周一
```

#### 标题识别逻辑

```
1. 文件名提取
   - 移除日期部分和扩展名
   - "2024-04-01_product_strategy_meeting.txt" → "product_strategy_meeting"
   - 下划线替换为空格，首字母大写: "Product Strategy Meeting"
   
2. 内容提取
   - 取文件第一行（去除时间戳）
   - "[10:30] 产品策略会" → "产品策略会"
   
3. 优先级: 内容提取 > 文件名提取
4. 如果都提取不到，显示为 "未命名会议"
```

### 7.2 说话人识别算法

#### 配置匹配逻辑

```
输入: 
- 用户常用身份列表: ["我", "小美", "产品经理-小美"] (按使用频率排序)
- 会议文件解析出的所有说话人: ["张三", "李四", "我", "王五"]

匹配算法:
1. 遍历常用身份列表（按优先级）
2. 对每个常用身份，检查是否在会议说话人列表中
3. 第一个匹配到的作为默认选项
4. 如果没有匹配到，使用"从文件识别"策略

从文件识别策略:
1. 统计每个说话人的发言次数
2. 使用第一人称("我认为", "我觉得") 频率最高的话术人
3. 如果没有明显特征，选择发言次数最多的作为推荐
4. UI 显示: "从本次文件识别... (推荐: '我')"
```

### 7.3 长文本分块处理算法

#### 分块策略

```
输入: 原始会议文本, 模型最大上下文限制 (如 128K tokens)

分块步骤:
1. 估算总 token 数
   - 中文: 字符数 × 1.2 (保守估计)
   - 英文: 字符数 × 0.3
   - 混合: 按实际字符类型分别计算

2. 判断是否需分块
   - 阈值: 模型限制 × 0.7 (预留 30% 给输出和系统提示)
   - 如果总 token < 阈值: 单块处理
   - 否则: 进入分块流程

3. 分块流程
   
   步骤 1: 按发言分割
   - 将文本按发言段落分割
   - 每个发言是一个不可分割的原子单元
   
   步骤 2: 贪心分组
   - 目标: 每组 token 数 ≈ 阈值
   - 策略: 按发言顺序累加，直到接近阈值
   - 边界处理: 如果单个发言超过阈值，按句子分割
   
   步骤 3: 上下文重叠
   - 每块包含前一块的最后 200 tokens 摘要
   - 确保话题连贯性
   
   示例:
   原文: [发言1, 发言2, 发言3, 发言4, 发言5, 发言6]
   假设每组可容纳 2 个发言
   
   块1: [发言1, 发言2]
   块2: [发言2摘要, 发言3, 发言4]
   块3: [发言4摘要, 发言5, 发言6]

4. 并行处理
   - 各块独立调用 LLM
   - 收集所有结果后合并
   - 注意: 话题标签需要去重和一致性处理

5. 结果合并
   - 发言数据: 直接拼接
   - 话题标签: 跨块统一命名（如"产品策略"和"产品方案"合并）
   - 关联关系: 基于完整数据重新计算
```

#### 分块数量与模型选择建议

```
| 文件大小 | Token 估算 | 建议模型 | 分块数 | 预估成本 |
|----------|-----------|----------|--------|----------|
| < 10K 字 | < 12K | Kimi 128K | 1 | ¥0.1-0.2 |
| 10-30K 字 | 12-36K | Kimi 128K | 1-2 | ¥0.2-0.5 |
| 30-80K 字 | 36-96K | Kimi 128K | 2-4 | ¥0.5-1.2 |
| 80-150K 字 | 96-180K | Gemini 1.5 Pro 或分 4-8 块 | 4-8 | ¥1.2-2.5 |
| > 150K 字 | > 180K | 建议拆分文件 | - | - |
```

### 7.4 话题提取算法

#### 自动话题标签生成

```
输入: 清理后的发言文本
输出: 话题标签列表 (1-3 个)

算法 1: LLM 提取 (主要方式)
- Prompt 模板: "请为以下发言提取 1-3 个话题标签，标签应简洁（2-4字），使用名词短语"
- 返回格式: JSON 数组 ["话题1", "话题2"]
- 示例: "我同意张三的观点。此外，我认为我们还需要考虑成本问题..."
  → ["产品策略", "成本管理"]

算法 2: 关键词匹配 (备用方式，当 LLM 不可用时)
- 预定义话题词库: {"产品": ["产品", "需求", "功能"], "技术": ["架构", "代码", "系统"]}
- TF-IDF 计算关键词权重
- 与词库匹配，取 Top 3

标签标准化处理:
1. 同义词合并: "产品策略" = "产品方案" = "产品规划"
2. 层级归一: "用户增长策略" → "用户增长"
3. 停用词过滤: 去除"会议"、"讨论"等通用词
```

#### 话题关联度计算

```
知识图谱中，话题之间的关联线粗细（关联强度）计算:

1. 共现关联 (Co-occurrence)
   - 定义: 两个话题在同一场会议中同时出现的次数
   - 计算: co_occurrence_count / total_meetings
   - 权重: 0.4 (基础关联)

2. 时间关联 (Temporal)
   - 定义: 两个话题在相近时间被讨论
   - 计算: 1 / (时间差天数 + 1)
   - 如果话题 A 在 04-01, 话题 B 在 04-03 → 时间关联 = 1/3
   - 权重: 0.2 (辅助关联)

3. 语义关联 (Semantic)
   - 定义: 话题内容的语义相似度
   - 计算: 使用词向量或 LLM 计算余弦相似度
   - 相似度 > 0.7 视为相关
   - 权重: 0.4 (强关联)

4. 综合关联度公式:
   
   association_score = (
       0.4 × normalize(co_occurrence) +
       0.2 × temporal_score +
       0.4 × semantic_score
   )
   
   阈值判断:
   - score >= 0.7: 强关联 (粗线)
   - 0.4 <= score < 0.7: 中关联 (中线)
   - 0.2 <= score < 0.4: 弱关联 (细线)
   - score < 0.2: 无关联 (不显示线)

5. 关联原因说明 (用于 UI 提示)
   - 当用户点击关联线时，显示关联原因:
   - "产品策略" 和 "用户增长":
     - 在 5 场会议中同时出现
     - 最近一次都在 2024-04-01 讨论
     - 语义相似度 82%
```

### 7.5 知识图谱布局算法

#### 混合布局（话题聚类 + 时间排列）

```
目标: 将话题以"岛屿"形式展示，岛屿内部按时间排列

步骤 1: 话题聚类 (岛屿形成)
- 每个话题是一个岛屿的中心
- 岛屿大小 = sqrt(该话题下发言数) × 缩放系数
- 岛屿位置: 使用力导向算法初始化
  - 节点间斥力: 防止重叠
  - 关联引力: 有关联的话题相互吸引

步骤 2: 岛屿内部时间排列
- 每个岛屿内部是一个垂直时间轴
- Y 轴: 时间（上旧下新）
- X 轴: 话题内子分类（如果有）
- 发言节点按时间戳垂直排列
- 相邻发言用细线连接显示时序

步骤 3: 力导向优化
- 定义三种力:
  1. 斥力 (Coulomb): 所有节点间相互排斥，防止重叠
  2. 引力 (Hooke): 
     - 话题内相邻发言相互吸引
     - 有关联的话题中心相互吸引
  3. 中心引力: 所有节点向画布中心轻微吸引，防止散落

步骤 4: 碰撞检测与调整
- 检测岛屿间重叠
- 如果重叠，沿连线方向推开，保持关联可见

步骤 5: 视口适应
- 计算所有节点的包围盒
- 自动缩放和平移，使所有节点在初始视口内可见

动画策略:
- 新增节点: 从中心放大出现
- 位置更新: 平滑过渡 (300ms ease-out)
- 关联线: 渐显效果
```

#### 布局参数

```yaml
layout_params:
  island:
    min_radius: 60        # 最小岛屿半径 (px)
    max_radius: 200       # 最大岛屿半径 (px)
    padding: 20           # 岛屿内边距 (px)
    
  node:
    speech_radius: 6      # 发言节点半径 (px)
    topic_font_size: 16   # 话题标题字号 (px)
    
  force:
    repulsion: 1000       # 斥力系数
    attraction: 0.05      # 引力系数
    damping: 0.9          # 阻尼系数 (用于动画)
    
  time_axis:
    spacing: 30           # 时间轴节点间距 (px)
    direction: "vertical" # 时间轴方向
```

## 8. 技术实现规范

### 8.1 话题关联语义相似度实现

**决策**: 使用 Embedding API + 缓存策略

**实现细节**:

```
1. Embedding 计算流程
   
   新话题入库时:
   - 提取话题名称（如"产品策略"）
   - 调用 Embedding API 获取向量（1536维）
   - 存储到 topic_embeddings 表
   
   话题关联计算时:
   - 从新话题出发，计算与所有已有话题的 cosine 相似度
   - 公式: similarity = dot_product(v1, v2) / (||v1|| * ||v2||)
   - 只计算一次，结果存入 topic_relations 表
   - 阈值: > 0.75 视为语义相关

2. 缓存策略
   
   数据库表结构:
   - topic_embeddings: topic_id, embedding_vector, created_at
   - topic_relations: topic_a_id, topic_b_id, similarity_score, created_at
   
   更新触发:
   - 新增话题时，计算与所有已有话题的关联
   - 已有话题关联不重新计算（增量更新）

3. API 成本控制
   
   - 单用户话题数通常 < 50
   - 新增 1 个话题最多调用 1 次 Embedding API
   - 关联计算使用本地向量运算，无额外 API 成本
   - 预估: 每新增 10 个话题，消耗约 ¥0.01-0.02
```

### 8.2 知识图谱布局实现

**决策**: 增量布局，保持已有话题位置稳定

**实现细节**:

```
1. 布局数据存储
   
   数据库表结构:
   - graph_layouts: 
     - user_id
     - layout_data: JSONB 存储所有节点坐标
     - version: 布局版本号
     - updated_at

2. 增量布局算法
   
   新话题加入时:
   a. 找出与之关联度最高的已有话题（引力中心）
   b. 在引力中心周围寻找空位（螺旋搜索）
   c. 距离 = 基础距离 / 关联度（关联越强，距离越近）
   d. 检测碰撞，如有重叠则微调位置
   
   螺旋搜索算法:
   ```
   angle = 0
   radius = base_radius
   while collision_detected:
       x = center_x + radius * cos(angle)
       y = center_y + radius * sin(angle)
       if not check_collision(x, y, new_topic_size):
           return (x, y)
       angle += 0.5  # 30度步进
       radius += 5   # 逐步外扩
   ```

3. 前端渲染
   
   - 首次加载: 从后端获取完整布局数据
   - 增量更新: WebSocket 推送新节点位置和关联
   - 动画: 新节点从中心放大出现（300ms ease-out）
   - 拖拽: 用户可手动调整节点位置，保存到后端
```

### 8.3 长文本分块一致性实现

**决策**: 分块处理完后，统一调用 LLM 进行标签标准化

**实现细节**:

```
1. 分块处理流程
   
   第 1 阶段 - 分块并行处理:
   - 按发言分割文本
   - 每块独立调用 LLM 提取标签
   - 块1 → ["产品策略", "成本管理"]
   - 块2 → ["产品方案", "用户增长"]
   
   第 2 阶段 - 标准化后处理:
   - 收集所有块提取的原始标签
   - 统一调用 LLM 进行标准化:
     
     Prompt: """

     输入标签: ["产品策略", "成本管理", "产品方案", "用户增长"]
     
     规则:
     1. 同义词合并（如"产品策略"和"产品方案"）
     2. 使用最通用的表达作为标准名
     3. 返回映射关系: {"原标签": "标准标签"}
     
     输出格式: {"产品策略": "产品策略", "产品方案": "产品策略", "成本管理": "成本管理", "用户增长": "用户增长"}
     """
   
   第 3 阶段 - 标签替换:
   - 根据映射关系，统一替换所有发言的标签
   - 更新 topic 表，合并重复话题

2. 一致性保证
   
   - 同一会议的所有发言在标准化后话题一致
   - 跨会议的话题通过 Embedding 相似度判断是否合并
   - 用户可手动调整标签映射关系
```

### 8.4 实时处理实现

**决策**: Server-Sent Events (SSE) + 任务队列

**实现细节**:

```
1. 架构组件
   
   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
   │   Web App   │◀───▶│   API Server│◀───▶│ Task Queue  │
   │   (React)   │ SSE │   (FastAPI) │     │  (Celery)   │
   └─────────────┘     └─────────────┘     └──────┬──────┘
                                                  │

                                           │   (Kimi)    │

   a. 用户上传文件 → API Server 接收
   b. 创建任务记录 (status=pending)
   c. 提交到 Celery 队列

4. 并发控制
   
   - 单用户并发限制: 最多 3 个文件同时处理
   - 全局队列长度限制: 防止系统过载
   - 任务超时: 单文件 10 分钟，超时自动失败
```

### 8.5 数据存储实现

**决策**: 使用 PostgreSQL 纯关系数据库，JSONB 存储图数据

**实现细节**:

```
1. 数据库选型理由
   
   - 个人用户数据量小（< 1000 会议，< 100 话题）

   - 图谱只是可视化，不需要复杂图查询
   - PostgreSQL JSONB 足够存储和查询图结构
   - 减少运维复杂度（单一数据库）

| API Key 无效 | "API Key 验证失败" | "请检查模型配置中的 API Key" |
| 配额不足 | "本月处理额度已用完" | "查看配额管理或升级计划" |

### 8.2 错误提示设计原则

1. **用用户的语言**：不说"ParseError"，说"无法识别文件格式"
2. **提供上下文**：告诉用户发生了什么，为什么会发生
3. **明确的下一步**：始终提供可操作的解决方案
4. **保留数据**：处理失败时保留已完成的步骤，支持断点续传

### 8.3 错误提示示例

```
❌ 处理中断

会议文件：product_meeting_0401.txt
中断位置：口语清理阶段（3/5 完成）

原因：与智能处理服务的连接超时

你可以：
  1. [重试处理] - 从断点继续（推荐）
  2. [跳过清理] - 使用原始文本继续
  3. [切换模型] - 使用已配置的备用模型

如果问题持续，请检查网络连接或联系支持。
```

## 10. 非功能需求

### 9.1 性能要求

| 场景 | 目标 | 说明 |
|------|------|------|
| 文件上传 | < 3s | 10MB 文件上传完成 |
| 解析处理 | < 5s | 1 小时会议内容解析 |
| 清理处理 | < 30s/块 | LLM 处理完成（每块约 8000 tokens）|
| 图谱渲染 | < 2s | 100 个节点以下的图谱 |
| 页面切换 | < 1s | 时间线↔图谱视图切换 |
| 搜索结果 | < 500ms | 关键词搜索响应 |

### 9.2 长文本处理策略

| 文件大小 | 处理策略 | 用户体验 |
|----------|----------|----------|
| < 8K tokens | 单块处理 | 快速完成，无感知 |
| 8K-32K tokens | 分 2-4 块并行处理 | 显示"分成 X 段处理" |
| 32K-100K tokens | 分 4-12 块串行处理 | 显示进度条和剩余段数 |
| > 100K tokens | 建议拆分文件 | 提示"文件过大，建议拆分" |

### 9.3 可用性要求

- **首次使用引导**：新用户 5 分钟内完成首次上传并看到结果
- **无障碍支持**：支持键盘导航，符合 WCAG 2.1 AA 标准
- **响应式设计**：支持 1280x720 及以上分辨率
- **浏览器兼容**：Chrome, Safari, Firefox, Edge 最新 2 个版本

### 9.4 安全与隐私

- **数据加密**：用户上传的文件和处理结果加密存储
- **访问控制**：用户只能访问自己的数据
- **API 安全**：API Key 加密存储，支持轮换
- **隐私模式**：支持"本地处理"选项，敏感数据不上传云端
