## 1. 背景与目标

### 1.1 问题陈述
会议语音转写文本存在以下问题：
- 口语化表达严重（"呃"、"啊"、"那个"等语气词过多）
- 语音识别可能产生错别字
- 缺乏结构，难以回顾和检索
- 无法直接用于 AI Agent 训练

### 1.2 项目目标
构建一个个人知识管理工具，实现：
- 自动提取个人发言内容
- 智能清理口语化表达和错别字，保留原意
- 提炼会议金句并书面化
- 记录时间戳，追踪思考演变
- 构建个人知识图谱，支持可视化探索
- 生成双轨输出（人读的 Markdown + AI 消费的 JSON）

### 1.3 成功标准
- [ ] 处理后文本通顺、无语病、保留原意
- [ ] 金句提炼准确，书面化表达符合语境
- [ ] 可按时间线查看思考演变
- [ ] JSON 格式可被外部 AI Agent 读取和解析
- [ ] 支持话题关联和知识图谱可视化

## 2. 用户故事

### 2.1 核心用户场景

**场景 1：发言记录与清理**
> 作为**会议参与者**，我希望**提取我在会议中的发言并清理口语**，以便**保留有价值的思考记录，方便后续回顾**。

**场景 2：思考演变追踪**
> 作为**知识管理者**，我希望**按时间追踪同一话题下我的观点变化**，以便**看到自己的思考演变和成长轨迹**。

**场景 3：AI Agent 语料积累**
> 作为**AI Agent 使用者**，我希望**将结构化数据喂给 AI**，以便**让 AI 更了解我的思维方式和表达习惯**。

## 3. 功能需求

### 3.1 功能列表

| 功能模块 | 需求描述 | 优先级 | 验收标准 |
|----------|----------|--------|----------|
| **发言提取** | 从转写文本中识别"我"的发言段落，区分其他参与者 | P0 | 准确识别说话人，不遗漏、不误判 |
| **口语清理** | 去除语气词（"呃"、"啊"、"那个"），修正错别字，保留原意 | P0 | 清理后文本通顺，不改变原意 |
| **金句提炼** | 从发言中提取核心观点，转化为书面化表达 | P0 | 金句准确概括核心观点，表达精炼 |
| **时间记录** | 记录每段发言的时间戳，支持时间线查看 | P0 | 时间戳准确，可按时序排列 |
| **话题标签** | 自动或手动为发言打标签，便于分类和检索 | P1 | 标签准确反映内容主题 |
| **知识图谱** | 可视化展示话题、发言、时间的关联关系 | P1 | 图谱清晰展示关联关系，支持交互探索 |
| **双轨输出** | 同时生成人读的 Markdown 和 AI 消费的 JSON | P0 | 两种格式都满足各自使用场景 |
| **批量处理** | 支持一次处理多个历史会议纪要文件 | P1 | 能批量导入并统一处理 |
| **增量更新** | 新会议纪要不覆盖历史数据，支持增量添加 | P1 | 新增数据与历史数据正确合并 |

### 3.2 优先级定义
- **P0 (Must Have)**: 核心功能，MVP 必须包含
- **P1 (Should Have)**: 重要功能，版本 1.0 应包含
- **P2 (Nice to Have)**: 增强功能，后续版本考虑

## 4. 数据需求

### 4.1 输入数据格式

**会议转写文本示例**:
```
[10:30:15] 张三：我觉得这个方案呃...可能还需要再考虑一下
[10:30:45] 我：我同意张三的观点。另外呢，我觉得我们还需要考虑成本问题...
[10:31:20] 李四：成本确实是个问题
```

**输入格式要求**:
- 支持纯文本 (.txt) 或 Markdown (.md)
- 每行包含时间戳和说话人标识
- 可配置时间戳和说话人格式

### 4.2 输出数据格式

**输出 1：人读 Markdown**
```markdown
# 会议纪要 - 2024-04-01 产品策略会

## 我的发言时间线

### 10:30:45 - 成本考量
**原文**: 我同意张三的观点。另外呢，我觉得我们还需要考虑成本问题...

**清理后**: 我同意张三的观点。此外，我认为我们还需要考虑成本问题...

**金句**: "成本效益分析是决策的关键维度，需要在方案设计初期纳入考量。"

**话题**: #产品策略 #成本管理 #决策
```

**输出 2：AI 消费 JSON**
```json
{
  "meeting": {
    "title": "产品策略会",
    "date": "2024-04-01",
    "participants": ["张三", "李四", "我"]
  },
  "my_speeches": [
    {
      "id": "speech_001",
      "timestamp": "10:30:45",
      "raw_text": "我同意张三的观点。另外呢，我觉得我们还需要考虑成本问题...",
      "cleaned_text": "我同意张三的观点。此外，我认为我们还需要考虑成本问题...",
      "key_quotes": [
        "成本效益分析是决策的关键维度，需要在方案设计初期纳入考量。"
      ],
      "topics": ["产品策略", "成本管理", "决策"],
      "sentiment": "neutral",
      "word_count": 45
    }
  ],
  "knowledge_graph": {
    "nodes": [...],
    "edges": [...]
  }
}
```

## 5. 非功能需求

### 5.1 性能要求
- 单份会议纪要（1小时）处理时间 < 30秒
- 支持批量处理最多 50 份文件

### 5.2 可用性要求
- 命令行界面简单易用
- 支持配置文件自定义参数
- 错误提示清晰，包含处理建议

### 5.3 安全与隐私
- 所有数据处理在本地完成（或可选本地 LLM）
- 不上传敏感会议内容到第三方（除非用户明确选择）
- 支持 API Key 加密存储

## 6. 技术方案

### 6.1 方案选择
**选用方案**: 方案 C（个人知识图谱）

### 6.2 技术栈建议
- **编程语言**: Python（推荐，AI/文本处理生态完善）
- **LLM 服务**: Claude API / OpenAI API（用于文本清理和金句提炼）
- **数据处理**: pandas, json
- **知识图谱**: networkx（基础）或 neo4j（进阶）
- **可视化**: D3.js 或 pyvis（知识图谱可视化）
- **CLI 框架**: typer 或 click

### 6.3 系统架构
```
输入层: 会议转写文本 (.txt/.md)
    ↓
处理层: 
  - 发言提取（正则/规则匹配）
  - 口语清理（LLM API）
  - 金句提炼（LLM API）
  - 话题标签（LLM 或关键词匹配）
    ↓
存储层:
  - 原始数据（JSON）
  - 知识图谱（图数据库/JSON）
    ↓
输出层:
  - Markdown 报告（人读）
  - JSON 数据（AI 消费）
  - 可视化图谱（HTML）
```

### 6.4 部署方式
- **阶段 1**: 本地 Python 脚本，命令行运行
- **阶段 2**: 可选 Web 界面，支持文件上传和可视化

## 7. 项目规划

### 7.1 里程碑

| 里程碑 | 交付物 | 时间估算 |
|--------|--------|----------|
| **MVP** | 发言提取 + 口语清理 + Markdown 输出 | 1 天 |
| **V1.0** | + 金句提炼 + JSON 输出 + 时间线 | 2-3 天 |
| **V1.5** | + 话题标签 + 批量处理 | 1-2 天 |
| **V2.0** | + 知识图谱 + 可视化 | 3-5 天 |

### 7.2 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM API 调用成本高 | 中 | 支持本地 LLM（如 Ollama）作为备选 |
| 发言识别准确率不足 | 高 | 允许用户手动校正，提供配置选项 |
| 话题标签不准确 | 中 | 结合自动+手动标签，支持用户调整 |
